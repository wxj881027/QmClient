# 页面切换性能优化设计文档

**日期**：2026-04-22  
**版本**：1.0  
**作者**：AI Assistant  
**状态**：设计完成，待实现

---

## 1. 概述

### 1.1 背景

QmClient 当前存在明显的设置页与页面切换卡顿，且已经通过多轮 `perf` 日志确认：这些卡顿并不是同一种问题，而是多个页面各自把不同类型的冷路径、派生路径或大规模 UI 构建塞进了主线程渲染热路径。

现有埋点与日志已经确认以下事实：

- 主线程阻塞已经成立。
- 主要卡点在 UI / 渲染主线程，不是输入轮询或事件风暴。
- `general`、`language`、`tclient`、`qmclient`、`assets` 五类页面的卡顿模式不同，不能用单一补丁处理。
- 现有埋点已足够支撑第一轮定向优化，不需要先做“大而全”重构。

### 1.2 目标

- 系统性降低设置页和页面切换的首帧卡顿与停留期慢帧。
- 把冷路径、派生路径、I/O 收尾和大规模不可见 UI 构建从 render 热路径中剥离。
- 在 `视觉无明显退化` 的前提下，以低风险、可验证、可回滚的方式分轮推进优化。
- 保留并复用现有 `perf` 埋点体系，使每一轮优化都能通过日志直接验证收益。

### 1.3 非目标

- 本轮设计不尝试统一重写整个设置系统。
- 本轮设计不主动改变 DDNet 兼容性相关逻辑。
- 本轮设计不以“现代化重构”为目标，不引入与现有架构风格不一致的大抽象。
- 本轮设计不优先追求极限帧时间，而是优先降低最明显的主线程尖峰。

## 2. 总体诊断

### 2.1 页面分类

当前页面可以分成 5 类性能问题：

1. **冷扫描 / 冷加载型**
   `general`
   典型表现是首次进入页面时，theme 枚举、目录扫描和 icon 加载落在 render 路径里。

2. **纯 UI / 文本容器型**
   `language`
   典型表现是列表构建和文本容器创建、渲染、销毁出现在热路径里。

3. **长滚动页全量构建型**
   `tclient`
   典型表现是屏幕外 section 仍然完整执行布局、控件逻辑和绘制，clip 只裁像素，不裁 CPU 工作。

4. **派生状态 + 模块同步渲染型**
   `qmclient`
   典型表现是 tab 切换时，同步做搜索匹配、列分配、模块可见性计算和热点模块渲染。

5. **异步任务主线程收尾型**
   `assets`
   典型表现是后台解码完成后，主线程在 render 中集中做 `ResizeImage(...)` 和 `LoadTextureRaw(...)`。

### 2.2 统一根因

虽然页面类型不同，但它们共享一个更底层的问题：

- 把冷路径放进 render。
- 把派生路径放进 render。
- 把屏幕外内容也完整渲染。
- 用“固定数量上限”代替“主线程时间预算”。

因此，总方案不应该只做单点 patch，而应该围绕以下统一框架展开。

## 3. 统一优化原则

### 3.1 原则一：页面级缓存 + 明确失效条件

所有可缓存的数据都需要回答两个问题：

- 缓存的对象是什么。
- 何时失效。

本次设计中，缓存对象主要包括：

- theme 元数据与 icon。
- 文本容器。
- tab 派生状态。
- 可见模块与列分配。
- 预览解码完成后的 ready 队列。

### 3.2 原则二：可见区优先

屏幕外内容不应完整执行重型 UI 逻辑。优化方向包括：

- section 级可视区裁剪。
- 模块级可视区裁剪。
- 可见区 + 邻近区预取。
- 首屏优先、非关键内容延后。

### 3.3 原则三：冷路径从 Render 剥离

以下工作默认都应从 render 热路径中剥离：

- 文件系统扫描。
- 目录枚举。
- 纹理或图标冷加载。
- 大规模文本容器重建。
- 搜索/排序/列分配等派生状态重算。
- 大图缩放与纹理上传。

### 3.4 原则四：毫秒预算优先于固定数量上限

固定“每帧最多处理 N 个对象”并不能约束主线程尖峰，因为：

- 单个对象成本可能差异极大。
- 少量大对象也能造成严重慢帧。

本设计默认优先使用：

- 每帧最多消耗 `X ms` 的时间预算。
- 在预算内处理尽可能多的 ready 项。
- 超预算立即停止，剩余项下一帧继续。

## 4. 路线选择

### 4.1 方案 A：保守型

特点：

- 不明显改变 UI 行为和画质。
- 先把冷路径、派生路径和 I/O 收尾移出 render 热路径。
- 以缓存、可视区优先、预算调度为主。

结论：

- 作为本项目的默认主路线。

### 4.2 方案 B：平衡型

特点：

- 在方案 A 基础上，再加入“首屏优先、非关键内容延后一帧或几帧补齐”。

结论：

- 不作为第一轮默认方案。
- 当某页面在完成方案 A 后仍存在明显首帧卡顿时，局部升级到方案 B。

### 4.3 方案 C：激进型

特点：

- 大量依赖后台预处理、延迟构建和更强的跨页缓存。

结论：

- 不纳入当前实现计划的第一阶段。

## 5. 页面级设计

### 5.1 `assets`

#### 5.1.1 已确认根因

- `merge/sort/search rebuild` 不是主因。
- 主因是后台解码完成后，主线程在 render 路径中集中执行：
  - `ResizeImage(...)`
  - `Graphics()->LoadTextureRaw(...)`
- 当前问题不是“列表管理慢”，而是“主线程预览缩放 + 上传”尖峰过大。

#### 5.1.2 第一轮优化

1. **可见区优先预览调度**
   - 只对当前可见卡片和邻近 1 至 2 行进行解码启动与 ready 处理。
   - 不再对搜索结果全集同步推进。

2. **ready queue + 毫秒预算**
   - 解码完成项进入 ready queue。
   - 每帧按时间预算处理队列头，而不是每帧全量扫描结果集。

3. **小改：使用 `LoadTextureRawMove`**
   - 在不改变画质的前提下减少一次图像拷贝。

#### 5.1.3 后续轮次

- 后台预缩放，把 `ResizeImage(...)` 从主线程移到 job 线程。
- 预览纹理跨 tab / 过滤复用缓存。

#### 5.1.4 风险

- 可见区优先会带来“滚动到新区域时，少量卡片先显示 loading”的短暂过渡。
- 该行为属于可接受范围，不视为视觉退化。

### 5.2 `general`

#### 5.2.1 已确认根因

- `RenderThemeSelection` 当前包含 theme 枚举、目录扫描和 icon 加载。
- 430 ms 级慢帧更像冷启动 / 冷加载型卡顿，而不是普通控件绘制过多。

#### 5.2.2 第一轮优化

1. **theme 元数据预热**
   - 在进入页面前或空闲帧预热 theme 列表。

2. **icon cache**
   - 页面只消费已就绪元数据。
   - icon 采用缓存，不在 render 路径中冷加载。

3. **可见区 / 邻近区 icon 懒加载**
   - 未就绪时使用占位，而不是阻塞页面首帧。

#### 5.2.3 后续轮次

- 引入目录版本或 `mtime` 缓存，避免不必要的全量重扫。

#### 5.2.4 风险

- icon 占位属于轻微视觉过渡，但不改变页面结构和最终表现。

### 5.3 `tclient`

#### 5.3.1 已确认根因

- 根因不是字体同步、滚动区起始或 section box 绘制。
- 根因是整页 section 在可视区裁剪前被完整执行。
- 左列最重，右列次之，`Tile outlines` 是明确热点但不是唯一来源。

#### 5.3.2 第一轮优化

1. **section 级可视区裁剪**
   - 每个 section 先计算外框并判断是否可见。
   - 屏幕外 section 只推进高度，不完整构建内部控件。

2. **重型控件降频**
   - 优先处理 `KeyReader`。
   - 减少非交互帧中的重复扫描与重复字符串比对。

3. **补足 section 级 perf 埋点**
   - 明确 `Visual`、`Player indicator`、`HUD`、`Tile outlines` 等 section 的真实占比。

#### 5.3.3 后续轮次

- 首屏优先，屏外重型 section 延后到次帧补齐。
- 对高级块做延迟构建或交互后构建。

#### 5.3.4 风险

- section 裁剪会改变“屏幕外内容也参与完整 UI 处理”的现有行为，但属于正确方向，不改变用户可见结果。

### 5.4 `qmclient`

#### 5.4.1 已确认根因

- 根因不是单一模块异常，而是：
  - tab 派生状态重算；
  - 搜索匹配；
  - 列分配；
  - 模块可见性计算；
  - 热点模块自身渲染；
  在 tab 切换首帧同步发生。

#### 5.4.2 第一轮优化

1. **tab 派生状态缓存**
   - 按 `tab/search/layout/order/collapse` 缓存：
     - visible modules
     - search 结果
     - 列分配
     - card metrics

2. **热点模块专项瘦身**
   - 第一轮只覆盖：
     - `pie_menu`
     - `mini_features`
     - `block_words`
     - `camera_view`
     - `entity_overlay`

3. **保持现有模块级 perf 埋点**
   - 作为回归验证依据。

#### 5.4.3 后续轮次

- 模块级可视区虚拟化。
- 首屏优先，屏外模块次帧补齐。

#### 5.4.4 风险

- 派生状态缓存的失效条件必须严格定义，否则容易出现 UI 与状态不同步。

### 5.5 `language`

#### 5.5.1 已确认根因

- 不是磁盘 I/O 问题。
- 主要是语言列表与 credits 文本容器的热路径成本。
- 当前 credits 文本容器仍存在每帧创建、渲染、销毁的模式。

#### 5.5.2 第一轮优化

1. **credits 文本容器缓存**
   - 仅在语言或宽度变化时重建。

2. **语言列表轻缓存**
   - 缓存选中项索引映射。
   - 缓存行级文本相关状态。

#### 5.5.3 后续轮次

- 非关键内容延后一帧补齐。

#### 5.5.4 风险

- 风险低，主要是缓存失效条件需要覆盖语言切换和宽度变化。

## 6. 分轮实施顺序

### 6.1 第 0 轮：框架与边界

目标：

- 明确各页面缓存对象与失效条件。
- 不改变页面行为。
- 只补最小必要埋点。

输出：

- 实施计划。
- section / queue / cache 失效规则。

### 6.2 第 1 轮：P0 页面

顺序：

1. `assets`
2. `general`
3. `tclient`

原因：

- 这三页的根因最明确。
- 第一轮就能直接打掉最大主线程尖峰。

### 6.3 第 2 轮：P1 页面

顺序：

1. `qmclient`
2. `language`

原因：

- 需要更多缓存边界设计，但仍然适合纳入统一框架。

### 6.4 第 3 轮：升级项

条件：

- 当方案 A 的收益不足以消除明显首帧卡顿时，局部升级到方案 B。

升级项：

- `assets` 后台预缩放。
- `tclient` 首屏优先。
- `qmclient` 模块级虚拟化。
- `language` 非关键内容延后。

## 7. 验证策略

### 7.1 验证方式

每轮优化都必须使用现有 `perf` 埋点和新日志回归验证。

验证维度：

- 页面首帧峰值是否下降。
- 停留期慢帧是否减少。
- 新增 stage 是否明确下降。
- 是否引入可见功能退化。

### 7.2 页面验收指标

#### `assets`

- `assets_preview_gpu_upload_scan` 峰值明显下降。
- `settings_page_content page=assets` 与 `frame_render` 同步尖峰明显减少。

#### `general`

- `theme_selection_total` 明显下降。

#### `tclient`

- `tclient_settings_left_column`、`tclient_settings_right_column` 明显下降。
- section 级埋点出现可解释分布。

#### `qmclient`

- `active_tab_total` 峰值下降。
- 热点模块 stage 下降或首帧分布被摊平。

#### `language`

- `language_page_total` 与 `language_list_total` 下降。
- credits 容器不再每帧重建。

## 8. 风险与边界

### 8.1 功能风险

- 缓存失效条件定义不完整，导致 UI 状态陈旧。
- 可视区裁剪处理不完整，导致屏幕外内容高度计算错误。
- ready queue 调度不当，导致资源预览加载过慢。

### 8.2 体验边界

允许：

- 占位图短暂存在。
- 预览渐进出现。
- 非关键内容下一帧补齐。

不允许：

- 明显画质退化。
- 页面布局变化。
- 交互功能缺失。
- 页面切换后内容长期不出现。

### 8.3 工程边界

- 第一轮不同时推进多个高风险重构。
- 每个页面优化都必须能单独验证、单独回滚。
- 不回退已有 perf 埋点。

---

## 9. 最终结论

本次整体优化采用以下主路线：

- **以方案 A 为主路线。**
- **只有在局部页面完成方案 A 后仍存在明显首帧卡顿时，才局部升级到方案 B。**

第一轮实现优先级固定为：

1. `assets`
2. `general`
3. `tclient`
4. `qmclient`
5. `language`

统一实施框架固定为：

1. 页面级缓存 + 明确失效条件。
2. 可见区优先。
3. 冷路径从 render 剥离。
4. 毫秒预算优先于固定数量上限。

这份设计的目标不是一次性重写设置系统，而是在现有 DDNet / QmClient 架构下，以最小必要改动系统性压低主线程尖峰。
