# 聊天翻译与聊天框翻译按钮设计

**日期：** 2026-04-21

**状态：** 已确认设计，待实现计划

## 目标

在不破坏现有聊天框自动隐藏行为的前提下，为 QmClient 增加一套面向聊天场景的翻译功能：

- 扩展现有翻译后端，支持机器翻译后端与通用 LLM API。
- 将智谱 AI 调整为通用 LLM 后端的预设，而不是单独架构。
- 在完整聊天框显示时提供一个全局翻译按钮，按钮弹出轻量行为设置面板。
- 支持收到消息自动翻译到用户选定语言。
- 支持自己输入的中文消息自动翻译到用户设定语言后再发送。

## 非目标

- 本期不在聊天框按钮中暴露后端、API Key、endpoint、model 配置。
- 本期不实现“每条聊天消息悬停时出现翻译按钮”。
- 本期不修复 Windows 第三方输入法候选框显示问题，只要求不要额外扩大问题面。
- 本期不改动服务端聊天协议。

## 用户体验设计

### 聊天框显示规则

- 聊天框平时继续保持自动隐藏逻辑。
- 仅在以下场景显示完整聊天框：
  - 玩家正在输入聊天内容。
  - 玩家按下手动显示完整聊天框的绑定键。
- 只有完整聊天框显示时，才显示聊天翻译全局按钮。

### 聊天框鼠标规则

- 完整聊天框显示时，自动显示并解锁聊天框鼠标。
- 关闭完整聊天框时，恢复现有鼠标锁定逻辑。
- 鼠标解锁与锁定必须复用现有 `CChat::UnlockMouse` / `CChat::LockMouse` 机制，避免并存两套状态。

### 全局翻译按钮

- 翻译按钮为聊天输入区旁边的全局按钮，不绑定到单条消息。
- 按钮只在完整聊天框可见时出现。
- 点击后弹出轻量设置面板。
- 面板只提供“聊天翻译行为设置”，不提供后端参数输入。

### 轻量设置面板内容

- 自动翻译总开关。
- 是否自动翻译收到的消息。
- 收到消息的目标语言。
- 是否自动翻译自己输入的消息后发送。
- 自己输入消息的目标语言。

## 翻译功能设计

### 入站翻译

- 入站消息自动翻译受“自动翻译总开关”和“自动翻译收到的消息”两个条件共同控制。
- 入站自动翻译范围限定为“玩家聊天消息”。
- 自动翻译继续跳过：
  - 客户端自身生成的消息。
  - 服务器消息。
  - 本地玩家自己发送的消息。
- 现有“检测到源语言为中文时可隐藏自动翻译结果”的兼容策略需要保留，但是否隐藏应与新的行为配置保持一致，不得与用户设置冲突。

### 出站翻译

- 出站自动翻译受“自动翻译总开关”和“自动翻译自己输入的消息后发送”两个条件共同控制。
- 触发条件为：
  - 输入内容包含中文字符。
  - 输入内容不是命令（即不以 `/` 开头）。
- 命中条件后，发送前先调用翻译后端，翻译成功则发送翻译结果，不再额外发送原文。
- 翻译失败时：
  - 不自动发送原文。
  - 在聊天中以客户端消息形式提示失败原因。
- 提供显式单次出站翻译命令 `/trans [text]`：
  - 该命令在聊天输入框中输入。
  - 命令命中后由客户端本地拦截，不作为普通聊天原文发送。
  - 翻译成功后发送翻译结果。
  - 发送目标沿用当前“出站目标语言”配置。
  - 发送频道沿用当前聊天模式，即公共聊天或队伍聊天。
- `/trans [text]` 与自动翻译开关并存：
  - 自动翻译用于日常输入。
  - `/trans [text]` 用于单次强制翻译发送。
- 现有尾部语言标记能力（如 `[en]`、`[ja]`）本期不再作为主交互设计，避免与 `/trans [text]` 和自动翻译开关形成三套并行入口。

### 目标语言

- 入站目标语言与出站目标语言分离配置。
- 两者都使用短语言代码字符串存储，例如 `zh`、`en`、`ja`、`zh-TW`。

## 后端架构设计

### 非 LLM 后端

保留以下现有机器翻译后端：

- TencentCloud
- LibreTranslate
- FTAPI

### 通用 LLM 后端

新增“通用 LLM 后端”抽象，配置包含：

- endpoint
- api key
- model

请求接口使用 OpenAI-compatible `chat/completions` 风格。

### 智谱 AI 预设

- 智谱 AI 不再作为单独后端架构存在。
- 智谱 AI 作为通用 LLM 后端的一个预设，提供默认：
  - endpoint
  - model
  - 翻译提示词模板

### 提示词

- 通用 LLM 后端支持内置提示词模板。
- 需要支持用户自定义提示词，以便优化游戏术语翻译。
- 智谱 AI 预设默认使用内置提示词，但允许被用户自定义提示词覆盖。
- 提示词要求模型“只输出翻译结果，不输出解释”。

## 配置设计

配置分为两层：

### 后端配置

保留在原设置页：

- 翻译后端选择
- endpoint
- API Key / Secret
- region
- model
- 通用 LLM / 智谱 AI 相关提示词配置

### 聊天行为配置

由聊天框翻译按钮弹出面板控制：

- 自动翻译总开关
- 入站自动翻译开关
- 入站目标语言
- 出站自动翻译开关
- 出站目标语言

这些配置需要持久化保存。

## 聊天消息渲染设计

- 入站翻译成功时：
  - 第一行显示翻译结果。
  - 若后端返回源语言，则在翻译结果后显示语言标记。
  - 第二行以较小字号显示原文。
- 翻译失败时：
  - 第一行显示原文。
  - 第二行以较小字号显示失败信息。
- 该渲染策略继续复用现有聊天容器和文本布局逻辑，不引入第二套聊天渲染实现。

## 错误处理

- 后端创建失败时，输出明确的客户端消息。
- HTTP 返回非 200 或 JSON 结构不符合预期时，输出后端名称、目标语言与错误摘要。
- 出站翻译失败不得悄悄吞掉消息；必须向用户显示失败原因。
- 配置非法时，优先阻止发起请求，并返回本地错误。

## 风险与边界

- 聊天框鼠标显示与输入法行为共用同一套交互状态，修改时必须避免引入新的焦点问题。
- 聊天框是高频路径，UI 改动需要尽量局部，避免影响非翻译用户。
- 通用 LLM 接口兼容范围以 OpenAI-compatible `chat/completions` 为准，不承诺兼容所有非标准扩展字段。

## 验收标准

- 用户可在完整聊天框中看到全局翻译按钮。
- 点击按钮后可调整聊天翻译行为设置。
- 入站自动翻译能按目标语言显示翻译结果。
- 出站自动翻译能在检测到中文输入时先翻译再发送。
- 用户可通过 `/trans [text]` 单次强制翻译并发送消息。
- 智谱 AI 通过通用 LLM 预设工作，不再依赖单独后端分支作为主架构。
- 原设置页继续负责后端与凭证配置，聊天框面板不暴露这些参数。
