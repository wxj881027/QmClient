# QmClient 页面切换卡顿与崩溃优化调研报告

## 一、问题概述

QmClient 在切换页面时存在明显的卡顿现象，且容易引发崩溃。本报告从架构层面深入分析了卡顿的根因、崩溃的触发条件，以及现有异步基础设施的完整能力，最终给出优化方案。

## 三、卡顿根因分析

### 3.1 第 1 层：主线程同步文件 I/O + 图片解码（最大瓶颈）

| 模块 | 操作 | 影响 |
|------|------|------|
| `CSkins7::Refresh()` | 同步扫描+加载所有皮肤 PNG + GPU 上传 | 初始化时长时间阻塞 |
| `CCountryFlags::LoadCountryflagsIndexfile()` | 同步加载 200+ 国旗 PNG + GPU 上传 | 初始化时长时间阻塞 |
| `CBackground::LoadBackground()` | 同步加载地图文件 + 图片 | 页面切换时阻塞 |
| `CMenuBackground::LoadMenuBackground()` | 同步加载主题地图 + 图标 | 主题切换时阻塞 |
| `StartEntitiesDecode` / `StartAssetDecode` | 主线程 `LoadFileToBuffer()` 后再异步解码 | 资产页面卡顿 |
| `Storage()->ListDirectory()` | 主线程同步扫描目录 | 首次进入资产页卡顿 |
| `LoadWorkshopCache()` | 主线程同步读取 512KB + JSON 解析 | Workshop 页面卡顿 |

### 3.2 第 2 层：批量 GPU 纹理上传无节流

| 模块 | 操作 | 影响 |
|------|------|------|
| `CSkins::LoadSkinFinish()` | 单个皮肤一次性上传 28+ GPU 纹理 | 多个皮肤完成时单帧 GPU 过载 |

| `CCountryFlags` | 每个国旗上传 1 个纹理 | 200+ 国旗同时上传 |

**对比**：资产设置页面已有节流机制（`MaxGpuUploadsPerFrame = 30`），但其他模块没有。

## 四、线程安全深度分析：官方为什么在后台线程调用 Graphics()？

### 4.1 官方设计意图

DDNet 官方的皮肤加载系统采用**经典的两阶段生产者-消费者模式**：

```
[后台线程]                              [主线程]
    │                                       │
    ├─ Graphics()->LoadPng() ──────────────→│  (PNG 解码为 RGBA 像素)
    │                                       │
    ├─ LoadSkinData() ─────────────────────→│  (像素验证、灰度转换、颜色计算)
    │                                       │
    │   [Job Done, 原子状态变更] ───────────→│
    │                                       ├─ UpdateFinishLoading() 检测 Done()
    │                                       ├─ LoadSkinFinish() 消费 m_Data
    │                                       │   └─ Graphics()->LoadSpriteTexture() 上传 GPU
    │                                       ├─ m_pLoadJob = nullptr (释放像素内存)
```

**为什么这样设计？** 因为 GPU API（OpenGL/Vulkan）的纹理上传操作**必须在拥有图形上下文的线程中执行**。官方将工作分为：
- **后台线程**：PNG 解码 + CPU 密集的像素处理（验证、灰度、颜色计算）
- **主线程**：仅执行 GPU 纹理上传（无法避免）

### 4.2 Graphics()->LoadPng() 的线程安全性详细分析

`CGraphics_Threaded::LoadPng()` 访问了以下共享状态：

| 共享状态 | 类型 | 访问方式 | 线程安全措施 |
|---|---|---|---|
| `m_WarnPngliteIncompatibleImages` | `std::atomic<bool>` | 只读 | **原子变量**，天然线程安全 |
| `m_vWarnings`（通过 `AddWarning`） | `std::vector<SWarning>` | 写入 | **`m_WarningsMutex` 互斥锁**保护 |
| `m_pStorage` | `IStorage*` 指针 | 调用 `OpenFile()` | 无锁，但初始化后不再变更 |

`AddWarning()` 和 `CurrentWarning()` 共享同一把 `m_WarningsMutex`，实现了生产者-消费者模式：

```cpp
void CGraphics_Threaded::AddWarning(const SWarning &Warning)
{
    const std::unique_lock<std::mutex> Lock(m_WarningsMutex);
    m_vWarnings.emplace_back(Warning);
}

std::optional<SWarning> CGraphics_Threaded::CurrentWarning()
{
    const std::unique_lock<std::mutex> Lock(m_WarningsMutex);
    if(m_vWarnings.empty()) return std::nullopt;
    auto Result = std::make_optional(m_vWarnings[0]);
    m_vWarnings.erase(m_vWarnings.begin());
    return Result;
}
```

**结论：`Graphics()->LoadPng()` 在后台线程调用是事实安全的。**

- `CImageLoader::LoadPng()` 是无状态的纯静态函数，天然线程安全
- `m_WarnPngliteIncompatibleImages` 使用 `std::atomic<bool>`
- `AddWarning()` 使用 `m_WarningsMutex` 保护
- `IStorage->OpenFile()` 在只读场景下事实安全（成员变量初始化后不可变，每次返回独立文件句柄）

### 4.3 IStorage 的线程安全性

`CStorage` 的 `ReadFile()` 和 `OpenFile()` **没有锁保护**，但事实安全：

| 层级 | 安全性 | 原因 |
|------|--------|------|
| CStorage 成员变量 | 事实安全 | `m_aaStoragePaths`、`m_NumPaths` 初始化后不再修改 |
| CStorage::OpenFile | 事实安全 | 只读成员变量拼接路径，调用 `io_open` |
| CStorage::ReadFile | 事实安全 | `OpenFile` 获取独立句柄 + `io_read_all` + `io_close` |
| io_open / fread / fclose | 线程安全 | 操作系统 API 保证，不同 `FILE*` 互不干扰 |
| Windows CreateFileW | 线程安全 | 使用 `FILE_SHARE_READ` 共享模式 |

**注意**：这是**事实上的线程安全**，而非**设计上的线程安全**。没有锁、没有文档保证、没有 `const` 限定。如果未来修改 CStorage 实现（如添加缓存），线程安全性可能被破坏。

### 4.4 m_Data 的数据传递安全性

`CSkinLoadData` 是后台线程与主线程之间的数据传输对象 (DTO)：

```cpp
class CSkinLoadData {
    CImageInfo m_Info;           // 原始 RGBA 像素数据
    CImageInfo m_InfoGrayscale;  // 灰度化后的像素数据
    CSkin::CSkinMetrics m_Metrics; // 皮肤尺寸度量
    ColorRGBA m_BloodColor;      // 血液颜色
};
```

线程安全保证：
1. `m_Data` 只在后台线程写入（`Run()` 中），只在主线程读取（`UpdateFinishLoading()` 中）
2. 通过 `IJob::Done()` 的原子状态检查作为同步点——`Done()` 返回 true 意味着 `Run()` 已完成，`m_Data` 不再被后台线程修改
3. IJob 文档明确说明："Accessing jobs in any other way than with the base functions of `IJob` is generally not thread-safe unless the job is in STATE_DONE or has not been enqueued yet."

### 4.5 官方设计是否应该改？

**短期：不需要改，但应添加防御性断言。**

官方的 `Graphics()->LoadPng()` 在后台线程调用是事实安全的，原因：
- `LoadPng()` 内部调用的 `CImageLoader::LoadPng()` 是纯 CPU 静态函数
- 共享状态访问都有原子变量或互斥锁保护
- `IStorage` 的只读操作事实安全
- `m_Data` 的数据传递通过 `IJob::Done()` 原子状态保证 happens-before 语义

**但存在以下隐患**：

1. **脆弱性**：如果未来有人在 `LoadPng()` 中添加了操作 CommandBuffer 的代码，将立即导致崩溃。当前 `LoadPng()` 不提交 GPU 命令，但这个约束没有任何代码级别的保证。

2. **`LoadSkinData()` 调用 `Graphics()->CheckImageDivisibility()` 和 `Graphics()->IsImageFormatRgba()`**：这两个方法只操作 `CImageInfo` 的 CPU 数据，不涉及 GPU，但同样通过 `Graphics()` 实例调用，访问了共享状态（`AddWarning()`）。

3. **缺乏显式文档**：没有任何注释说明 `LoadPng()` 可以在后台线程调用，也没有线程安全契约。

**长期：建议统一到 `CAssetDecodeJob` 模式。**

`CAssetDecodeJob` 的模式更安全、更清晰：

```
官方模式 (CSkinLoadJob):
  后台线程: Graphics()->LoadPng() → LoadSkinData()
  主线程:   LoadSkinFinish() → GPU 上传

推荐模式 (CAssetDecodeJob):
  主线程:   Storage()->ReadFile() → 读取文件数据到内存
  后台线程: CImageLoader::LoadPng(data, size) → 纯 CPU 解码
  主线程:   Graphics()->LoadTextureRaw() → GPU 上传
```

推荐模式的优势：
- 后台线程**完全不接触 `Graphics()` 实例**，消除对共享状态的依赖
- 使用 `CImageLoader::LoadPng()` 静态函数，无共享状态，天然线程安全
- 文件 I/O 和图片解码在同一次 Job 中完成，避免主线程的 `LoadFileToBuffer` 阻塞
- 更容易添加线程安全断言（如 `dbg_assert` 检查是否在主线程）

### 4.6 CSoundLoading 的特殊问题

```cpp
void CSoundLoading::Run()
{
    for(int s = 0; s < g_pData->m_NumSounds; s++)
        for(int i = 0; i < g_pData->m_aSounds[s].m_NumSounds; i++)
        {
            int Id = m_pGameClient->Sound()->LoadWV(pLoadFilename);  // 后台线程！
            if(m_Render)
                m_pGameClient->m_Menus.RenderLoading(...);  // 后台线程调用渲染！
        }
}
```

- `Sound()->LoadWV()` 在后台线程调用——音频驱动的线程安全性取决于实现
- `RenderLoading()` 在后台线程调用——**极其危险**，涉及 Graphics API
- 当 `g_Config.m_ClThreadsoundloading = true` 时，`m_Render = false`，不会调用 `RenderLoading()`
- 官方提供了 `m_ClThreadsoundloading` 开关，允许用户在同步/异步之间选择

## 五、页面切换卡顿的关键路径

### 5.1 页面切换流程

```
用户点击标签 → SetMenuPage(NewPage)
    ├── 触发切换动画（TriggerUiSwitchAnimation）
    ├── 如果切换到浏览器页面 → RefreshBrowserTab() → ServerBrowser()->Refresh()
    └── 渲染分发到对应 RenderXxx() 函数
```

### 5.2 各页面的卡顿热点

| 页面 | 首次加载操作 | 卡顿原因 |
|------|-------------|---------|
| **资产设置页** | `Storage()->ListDirectory()` + `LoadFileToBuffer()` | 同步目录扫描 + 文件读取 |
| **皮肤设置页** | `CSkins7::Refresh()` | 同步加载所有皮肤 PNG + GPU 上传 |
| **浏览器页** | `RefreshBrowserTab()` | 网络请求初始化 |
| **Demo 页** | `Storage()->ListDirectory()` | 同步目录扫描 |
| **背景切换** | `CMenuBackground::LoadMenuBackground()` | 同步地图/图片加载 |

### 5.3 资产设置页的加载流程（典型卡顿场景）

```
1. 首次进入资产页:
   Storage()->ListDirectory("assets/entities", EntitiesScan, &User)  ← 同步，阻塞主线程
   ↓
2. 对每个资产:
   LoadFileToBuffer(pStorage, aPath, IStorage::TYPE_ALL, vFileData)  ← 同步，阻塞主线程
   ↓
3. 提交异步解码 Job:
   pEngine->AddJob(CAssetDecodeJob(std::move(vFileData), ...))  ← 异步，不阻塞
   ↓
4. 主线程轮询完成状态:
   UploadCompletedDecodeJob()  ← 每帧最多上传 30 个纹理
```

**问题**：步骤 1 和 2 在主线程同步执行，如果资产数量多，会导致明显卡顿。

## 六、优化方案

### 6.1 方案 1：统一异步加载模式（最关键）

将所有同步加载改为 `CAssetDecodeJob` 模式——文件 I/O 和图片解码都在 Job 中完成，主线程只做 GPU 上传。

**需要重构的模块**：

| 模块 | 当前 | 目标 | 工作量 |
|------|------|------|--------|
| `StartEntitiesDecode` / `StartAssetDecode` | 主线程 `LoadFileToBuffer` | 文件读取移入 Job | 小 |
| `CSkins7::Refresh()` | 全同步 | 异步 Job + 主线程 GPU 上传 | 中 |
| `CCountryFlags` | 全同步 | 懒加载 + 异步 Job | 中 |
| `CBackground` / `CMenuBackground` | 全同步 | 异步 Job + 占位纹理 | 中 |
| `CSkinLoadJob` / `CSkinDownloadJob` | 后台调 `Graphics()` | 改用 `CImageLoader::LoadPng()` | 中 |
| `CCommunityIconLoadJob` | 后台调 `Graphics()` | 改用 `CImageLoader::LoadPng()` | 小 |

**统一后的模式**：

```cpp
class CAsyncImageLoadJob : public IJob {
    // 输入
    std::vector<uint8_t> m_vFileData;  // 主线程读取后移入
    std::string m_Name;

    // 输出
    std::mutex m_ResultMutex;
    CImageInfo m_ResultImage;
    bool m_Success = false;

    void Run() override {
        CImageInfo Image;
        bool Ok = CImageLoader::LoadPng(m_vFileData.data(), m_vFileData.size(),
                                         m_Name.c_str(), Image);
        std::lock_guard<std::mutex> Lock(m_ResultMutex);
        m_ResultImage = std::move(Image);
        m_Success = Ok;
    }
};

// 主线程使用
void LoadAsync(const char *pPath) {
    // 1. 主线程读取文件（或也移入 Job）
    std::vector<uint8_t> vData;
    Storage()->ReadFile(pPath, ..., &vData);

    // 2. 提交异步解码 Job
    auto pJob = std::make_shared<CAsyncImageLoadJob>(std::move(vData), pPath);
    Engine()->AddJob(pJob);
    m_PendingJobs.push_back(pJob);
}

void OnRender() {
    // 3. 主线程轮询完成状态，上传 GPU 纹理
    for(auto &pJob : m_PendingJobs) {
        if(!pJob->Done()) continue;
        if(GpuUploadsThisFrame >= MAX_GPU_UPLOADS_PER_FRAME) break;
        auto Result = pJob->GetResult();
        if(Result.m_Success)
            Texture = Graphics()->LoadTextureRaw(Result.m_Image, ...);
        pJob.reset();
    }
}
```

### 6.2 方案 2：全局 GPU 纹理上传节流

当前只有资产设置页面有节流（30/帧），其他模块没有。建议实现全局节流：

```cpp
class CGpuUploadLimiter {
    static constexpr int MAX_UPLOADS_PER_FRAME = 30;
    int m_UploadsThisFrame = 0;

    void OnFrameStart() { m_UploadsThisFrame = 0; }
    bool CanUpload() const { return m_UploadsThisFrame < MAX_UPLOADS_PER_FRAME; }
    void OnUploaded() { ++m_UploadsThisFrame; }
};
```

在 `CGameClient::OnRender()` 开头调用 `OnFrameStart()`，所有模块的 GPU 上传都通过 `CanUpload()` 检查。

### 6.3 方案 3：文件 I/O 也移入 Job

将 `LoadFileToBuffer()` 和 `Storage()->ReadFile()` 从主线程移入 Job 内部：

```cpp
class CFullAsyncImageLoadJob : public IJob {
    std::string m_Path;
    int m_StorageType;
    IStorage *m_pStorage;  // 事实线程安全

    void Run() override {
        // 1. 后台线程读取文件
        void *pData; unsigned Size;
        if(!m_pStorage->ReadFile(m_Path.c_str(), m_StorageType, &pData, &Size))
            return;
        // 2. 后台线程解码图片
        CImageInfo Image;
        m_Success = CImageLoader::LoadPng(static_cast<uint8_t*>(pData), Size,
                                          m_Path.c_str(), Image);
        free(pData);
        m_ResultImage = std::move(Image);
    }
};
```

**优势**：主线程完全不阻塞，文件 I/O 也在后台完成。
**风险**：`Storage()->ReadFile()` 在后台线程调用，虽然事实安全，但缺乏显式保证。

### 6.4 方案 4：线程安全加固（防御性措施）

无论是否重构加载模式，都应添加以下防御性措施：

1. **在 `CGraphics_Threaded` 中添加主线程断言**：

```cpp
bool CGraphics_Threaded::LoadPng(CImageInfo &Image, const char *pFilename, int StorageType)
{
    // 防御性断言：LoadPng 可以在后台线程调用，但 GPU 操作不行
    // 如果未来有人误在 LoadPng 中添加 GPU 操作，这个断言会捕获
    // 注意：当前 LoadPng 是纯 CPU 操作，此断言仅用于文档目的
    // dbg_assert(IsMainThread(), "LoadPng must be called from main thread");
    // ↑ 不建议启用，因为官方代码确实在后台线程调用
}
```

2. **在 `LoadTextureRaw` / `LoadSpriteTexture` 等 GPU 操作中添加主线程断言**：

```cpp
IGraphics::CTextureHandle CGraphics_Threaded::LoadTextureRaw(...)
{
    dbg_assert(IsMainThread(), "GPU texture upload must be called from main thread");
    // ...
}
```

3. **为 `CStorage` 添加线程安全文档注释**：

```cpp
// IStorage 接口
// 注意：ReadFile() 和 OpenFile(READ) 在初始化完成后可以从任意线程调用，
// 因为内部状态在初始化后不可变，且每次调用返回独立的文件句柄。
// 写入操作（OpenFile(WRITE)）不建议在后台线程调用。
```

## 七、实施优先级

| 优先级 | 任务 | 预期效果 | 工作量 |
|--------|------|---------|--------|
| **P0** | `StartEntitiesDecode` / `StartAssetDecode` 中 `LoadFileToBuffer` 移入 Job | 消除资产页面卡顿 | 小 |

## 八、结论

### 8.1 不需要新建 Job 池

`CJobPool` 已经存在且功能完善（线程数 = `max(4, hw_concurrency) - 2`），足以满足所有异步加载需求。

### 8.2 官方在后台线程调用 Graphics() 的设计是合理的

`Graphics()->LoadPng()` 虽然在后台线程调用，但：
- `CImageLoader::LoadPng()` 是纯 CPU 静态函数，天然线程安全
- 共享状态访问有原子变量或互斥锁保护
- `IStorage` 的只读操作事实安全
- 数据传递通过 `IJob::Done()` 原子状态保证 happens-before 语义

**短期不需要改**，但长期建议统一到 `CAssetDecodeJob` 模式（后台线程完全不接触 `Graphics()` 实例），以降低脆弱性。

### 8.3 真正需要改的是同步加载

页面切换卡顿的根因不是 Job 池不足，而是：
1. **大量操作没有使用现有的异步机制**（`CSkins7`、`CCountryFlags`、`CBackground` 全同步）
2. **部分异步 Job 的文件 I/O 仍在主线程**（`StartEntitiesDecode` 的 `LoadFileToBuffer`）
3. **GPU 纹理上传没有全局节流**（资产页面有 30/帧限制，但皮肤/国旗等没有）

修复这三点就能根本解决页面切换卡顿问题。
