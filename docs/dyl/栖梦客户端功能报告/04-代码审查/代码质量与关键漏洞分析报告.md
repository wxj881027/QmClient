# QmClient 代码质量与关键漏洞分析报告

## 一、概述

本报告对 QmClient 整个代码库进行了全面的代码质量审查，涵盖内存安全、线程安全、输入验证、资源管理和 QmClient 特有代码五个维度。共发现 **高危漏洞 12 个**、**中危漏洞 18 个**、**低危漏洞 10 个**。

## 二、高危漏洞（P0 — 必须修复）

### 2.1 语音模块 CRClientVoice — 多线程竞态条件（最严重）

**文件**: [voice_core.h](file:///e:/Coding/DDNet/QmClient/src/game/client/components/qmclient/voice_core.h) / [voice_core.cpp](file:///e:/Coding/DDNet/QmClient/src/game/client/components/qmclient/voice_core.cpp)

语音模块是项目中**线程安全问题最集中的组件**，涉及 Worker 线程、SDL 音频回调线程和主线程三者的并发访问，但大量共享状态没有适当的同步保护。

> 代码注释也明确标注了 `"Voice full wrote by AI don't use that pls"`（voice_core.cpp:41-42），表明此模块质量需要特别关注。

**漏洞 2.1.1 — m_aPeers 无锁竞态**

`m_aPeers`（`std::array<SVoicePeer, MAX_CLIENTS>`）在三个线程中被并发访问和修改，没有互斥锁保护：

- Worker 线程：`ProcessIncoming()` → `DecodeJitter()` 写入帧数据
- SDL 音频回调：`MixAudio()` 读写帧头/计数
- 主线程：`ClearPeerFrames()` / `ResetPeer()` 重置状态

`ClearPeerFrames()` 使用 `SDL_LockAudioDevice` 与音频回调同步，但 Worker 线程的 `DecodeJitter()` **完全不受此锁保护**。

**漏洞 2.1.2 — m_MixBuffer 竞态**

`std::vector<int32_t> m_MixBuffer` 在 Worker 线程的 `EnsureAudio()` 中 `resize()`，同时在 SDL 音频回调的 `MixAudio()` 中读写。如果 `resize()` 在回调使用期间触发，会导致悬空指针和内存损坏。

**漏洞 2.1.3 — 非原子共享变量**

以下变量在 Worker 线程和主线程之间共享，但**不是 `std::atomic`**：

```cpp
SDL_AudioDeviceID m_CaptureDevice = 0;  // Worker 和主线程都修改
SDL_AudioDeviceID m_OutputDevice = 0;   // Worker 和主线程都修改
bool m_VadActive = false;               // Worker 写，主线程读
int64_t m_VadReleaseDeadline = 0;       // Worker 写，主线程读
bool m_TxWasActive = false;             // Worker 写，主线程读
bool m_ShutdownDone = true;             // Worker 写，主线程读
```

对比同文件中正确使用 `std::atomic` 的变量：
```cpp
std::atomic<bool> m_PttActive = false;       // 正确
std::atomic<int64_t> m_PttReleaseDeadline = 0; // 正确
```

**漏洞 2.1.4 — OnRender/Worker 竞态**

`OnRender()` 中 `StopWorker()` + `ClearPeerFrames()` + `StartWorker()` 的序列存在竞态窗口。`m_WorkerStop.store(true)` 和 Worker 线程实际退出之间存在时间差。

### 2.2 整数溢出 — 导致内存分配不足

**漏洞 2.2.1** — [qmclient.cpp:5159](file:///e:/Coding/DDNet/QmClient/src/game/client/components/qmclient/qmclient.cpp#L5159)

```cpp
const int MapCellCount = Width * Height;  // 可能溢出！
```

同一文件第 4851 行对相同计算做了正确的防护：

```cpp
const int64_t MapSize64 = (int64_t)Width * Height;  // 安全写法
if(MapSize64 <= 0 || MapSize64 > std::numeric_limits<int>::max())
    return;
```

但第 5159 行没有使用相同防护，不一致。

**漏洞 2.2.2** — [backend_opengl.cpp:856](file:///e:/Coding/DDNet/QmClient/src/engine/client/backend/opengl/backend_opengl.cpp#L856)

```cpp
uint8_t *pImageData3D = static_cast<uint8_t *>(malloc((size_t)Width * Height * PixelSize));
```

`Height` 和 `PixelSize` 是 `int`，`(size_t)Width * Height` 中 `Height` 仍为 `int`，如果为负值则结果未定义。类似问题存在于 [backend_vulkan.cpp:2508](file:///e:/Coding/DDNet/QmClient/src/engine/client/backend/vulkan/backend_vulkan.cpp#L2508) 和 [backend_opengl3.cpp:657](file:///e:/Coding/DDNet/QmClient/src/engine/client/backend/opengl/backend_opengl3.cpp#L657)。

### 2.3 HTTP 明文通信 — 中间人攻击风险

**文件**: [qmclient.cpp:65-68](file:///e:/Coding/DDNet/QmClient/src/game/client/components/qmclient/qmclient.cpp#L65)

```cpp
static constexpr const char *QMCLIENT_HEALTH_URL = "http://42.194.185.210:8080/healthz";
static constexpr const char *QMCLIENT_PLAYTIME_START_URL = "http://42.194.185.210:8080/playtime/start";
// ... 所有 QmClient 同步请求都使用 HTTP + AllowInsecureProtocol()
```

**影响**：
- 所有 QmClient 生命周期同步、用户数据上报、auth token 获取都通过 HTTP 明文传输
- 多处调用 `AllowInsecureProtocol()` 允许降级为 HTTP
- 硬编码 IP 地址，服务器迁移时需要重新编译
- 中间人攻击者可以拦截并篡改 HTTP 响应，注入恶意 JSON 数据

### 2.4 密钥明文存储

**文件**: [config_variables_qmclient.h:260-262](file:///e:/Coding/DDNet/QmClient/src/engine/shared/config_variables_qmclient.h#L260)

```cpp

MACRO_CONFIG_STR(TcTranslateSecretKey, tc_translate_secret_key, 256, "", CFGFLAG_CLIENT | CFGFLAG_SAVE, ...)
```

### 2.5 内存泄漏 — json_value 错误路径

**文件**: [censor.cpp:136-148](file:///e:/Coding/DDNet/QmClient/src/game/client/components/censor.cpp#L136)

```cpp
if(pData->type != json_object)
{
```

### 2.6 命令注入 — 自动回复玩家名称

**文件**: [qmclient.cpp:1761-1763](file:///e:/Coding/DDNet/QmClient/src/game/client/components/qmclient/qmclient.cpp#L1761)

```cpp
str_format(aBuf, sizeof(aBuf), "/w %s %s", aPlayerName, g_Config.m_TcAutoReplyMutedMessage);
```

玩家名称 `aPlayerName` 直接拼接到 `/w` 命令中，**未加引号包裹**。如果玩家名称包含空格或特殊字符，可能导致命令注入。对比 [pie_menu.cpp:729](file:///e:/Coding/DDNet/QmClient/src/game/client/components/pie_menu.cpp#L729) 中使用了引号包裹：

```cpp
str_format(aBuf, sizeof(aBuf), "/w \"%s\" ", pPlayerName);  // 更安全
```

## 七、核心结论

1. **语音模块 (voice_core) 是最大风险点** — 8 个漏洞中 4 个高危，涉及三线程竞态、非原子变量、设备管理竞态。代码注释也标注了"AI 生成，不建议使用"。建议优先重构或添加全面的同步机制。

2. **HTTP 明文通信是最大安全风险** — QmClient 的所有中心服通信都使用 HTTP + `AllowInsecureProtocol()`，且硬编码 IP 地址。建议升级为 HTTPS + 域名 + 可配置 URL。

3. **密钥管理需要重新设计** — 腾讯云 SecretKey 和翻译 API Key 以明文保存在配置文件中，应改用操作系统密钥管理（如 Windows Credential Manager）。

4. **整数溢出是隐蔽的内存安全风险** — 多处 `Width * Height` 乘法未使用 `int64_t` 防护，可能导致分配不足后的缓冲区溢出。

5. **QmClient 特有代码的组件生命周期管理不完整** — 多个组件的 OnShutdown/OnReset 遗漏了资源清理，可能导致关闭时崩溃或内存泄漏。
