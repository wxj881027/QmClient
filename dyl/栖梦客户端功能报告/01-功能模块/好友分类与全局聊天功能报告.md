# QmClient 好友分类与全局聊天功能报告

## 一、好友分类系统

### 1.1 核心架构

好友系统由接口层和实现层两部分组成：

| 层级  | 类名         | 文件路径                                          |
| --- | ---------- | --------------------------------------------- |
| 接口层 | `IFriends` | `src/engine/friends.h`                        |
| 实现层 | `CFriends` | `src/engine/client/friends.h` / `friends.cpp` |

### 1.2 数据结构

**好友信息结构** (`CFriendInfo`):

```cpp
struct CFriendInfo
{
    char m_aName[MAX_NAME_LENGTH];       // 好友名称
    char m_aClan[MAX_CLAN_LENGTH];       // 好友战队
    char m_aCategory[64];                // 所属分类
    char m_aNote[128];                   // 好友备注
    unsigned m_NameHash;                 // 名称哈希（加速查找）
    unsigned m_ClanHash;                 // 战队哈希
};
```

**分类存储结构** (`CFriends`):

```cpp
CFriendInfo m_aFriends[MAX_FRIENDS];                              // 最多 4096 个好友
char m_aaCategories[MAX_FRIEND_CATEGORIES][MAX_FRIEND_CATEGORY_LENGTH]; // 最多 64 个分类
int m_NumFriends;
int m_NumCategories;
```

### 1.3 预置分类（受保护，不可删除/重命名）

| 常量                      | 值        | 说明       |
| ----------------------- | -------- | -------- |
| `DEFAULT_CATEGORY`      | `"好友"`   | 默认分类     |
| `CLAN_MEMBERS_CATEGORY` | `"战队成员"` | 战队成员自动归类 |
| `OFFLINE_CATEGORY`      | `"离线"`   | 离线好友自动归类 |

受保护分类的判断逻辑在 `CFriends::IsProtectedCategory`：

```cpp
bool CFriends::IsProtectedCategory(const char *pCategory)
{
    return pCategory != nullptr && (str_comp_nocase(pCategory, IFriends::DEFAULT_CATEGORY) == 0
        || IsClanMembersCategory(pCategory) || IsOfflineCategory(pCategory));
}
```

| 功能        | 控制台命令                                   | IFriends 方法                             | 实现状态  |

| 移动分类排序    | --                                      | `MoveCategory()`                        | ✅ 已实现 |

**添加分类**：新分类会插入到"离线"分类之前，保持离线分类始终在最后。

**删除分类**：删除分类时，该分类下的好友会自动移回默认分类（"好友"）。

**重命名分类**：重命名时会同步更新所有属于该分类的好友的 `m_aCategory` 字段。

### 1.5 UI 实现

好友列表的 UI 渲染在 `menus_browser.cpp` 中实现，核心要素包括：

- **CFriendItem**：好友列表项的 UI 数据类，包含名称、战队、分类、在线状态、皮肤信息等
- **分类折叠/展开**：通过 `m_vFriendsCategoryExpanded` 向量跟踪每个分类的展开状态，持久化到 `g_Config.m_ClFriendsCategoryExpanded`
- **分类拖拽排序**：支持拖拽分类头部重新排序
- **分类右键菜单**：右键分类头部弹出 PopupFriendsCategory，提供"新增分类"、"重命名"、"删除分类"三个操作
- **好友右键菜单**：右键好友弹出操作菜单，包含"移动到分类"选项
- **添加好友时选择分类**：通过下拉框 `m_FriendAddCategoryIndex` 选择分类

### 1.6 持久化

好友数据通过 `ConfigSaveCallback` 写入配置文件，格式如下：

```
friend_category_add "自定义分类名"
add_friend "玩家名" "战队名" "分类名"
set_friend_note "玩家名" "战队名" "备注内容"
```

## 二、聊天系统

### 2.1 当前聊天架构

聊天系统核心类为 `CChat`，位于 `chat.h` / `chat.cpp`。

### 2.2 聊天模式

| 模式   | 枚举值         | Team 参数                 | 说明        |
| ---- | ----------- | ----------------------- | --------- |
| 全体聊天 | `MODE_ALL`  | 0                       | 服务器内所有人可见 |
| 队伍聊天 | `MODE_TEAM` | 1                       | 仅同队玩家可见   |
| 发送私聊 | --          | 2 (`TEAM_WHISPER_SEND`) | 私聊发送方显示   |
| 接收私聊 | --          | 3 (`TEAM_WHISPER_RECV`) | 私聊接收方显示   |

### 2.3 消息处理流程

```
服务器 → CNetMsg_Sv_Chat → CChat::OnMessage() → CChat::AddLine() → 渲染/日志
```

### 2.4 私聊(Whisper)实现

私聊是**服务器端**功能，通过 `/w` 或 `/whisper` 命令触发。服务端使用 `TEAM_WHISPER_SEND` 和 `TEAM_WHISPER_RECV` 区分方向。客户端显示：发送方 `→ 玩家名`，接收方 `← 玩家名`。

### 2.5 QmClient 聊天扩展

| 组件         | 文件            | 功能           |
| ---------- | ------------- | ------------ |
| CBindChat  | bindchat.cpp  | 快捷聊天绑定系统     |
| CTranslate | translate.cpp | 聊天翻译功能       |
| 自动回复       | qmclient.cpp  | 最小化/被屏蔽时自动回复 |
| 屏蔽词        | chat.cpp      | 聊天消息屏蔽词过滤    |
| 聊天气泡       | chat.h        | 玩家头顶聊天气泡     |
| 草稿保存       | chat.cpp      | 保存未发送的聊天草稿   |
| 动画效果       | chat.cpp      | 消息淡入/滑入/淡出动画 |

## 三、全局聊天

### 3.1 当前状态：未实现 ❌

全局聊天（跨服务器聊天）功能**目前没有任何实现**。在整个 `src/` 目录中搜索 `globalchat`、`global_chat`、`cross-server`、`lobby chat` 等关键词，未找到任何相关代码。

### 3.2 实现全局聊天的技术障碍

1. **协议限制**：当前聊天完全依赖 `CNetMsg_Sv_Chat` / `CNetMsg_Cl_Say` 网络消息，这些消息仅在单个服务器连接内传输，没有跨服通信机制
2. **无中心化通信层**：客户端同时只连接一个游戏服务器，没有额外的通信通道用于跨服聊天
3. **CChat 扩展点有限**：
   - `AddLine()` 方法是 `private` 的，外部组件无法直接添加消息行
   - 聊天模式枚举 `MODE_ALL` / `MODE_TEAM` 是硬编码的
   - `SendChat()` / `SendChatQueued()` 仅发送到当前连接的服务器
4. **好友系统无唯一标识符**：当前好友系统仅基于名称+战队匹配，没有账号ID，跨服好友识别困难

### 3.3 可能的实现路径

| 方案            | 复杂度 | 可行性 | 说明                                               |
| ------------- | --- | --- | ------------------------------------------------ |
| 独立聊天服务器       | 高   | 中   | 需要搭建和维护额外服务，客户端需建立第二条连接                          |
| Discord Lobby | 中   | 中   | 项目已集成 Discord SDK，可利用 Lobby 消息功能，但需要用户安装 Discord |
| 游戏服务器中继       | 高   | 低   | 需要所有游戏服务器配合，改动范围大                                |
| HTTP 轮询       | 低   | 低   | 简单但延迟高，不适合实时聊天                                   |
